3:T679,<p>Animation performance is one of those topics that separates good frontend engineers from great ones. It is not about knowing every property — it is about understanding how the browser works.</p>
<h2>The Rendering Pipeline</h2>
<p>The browser renders a frame in roughly this order:</p>
<ol>
<li><strong>Style</strong>: Calculate which CSS rules apply</li>
<li><strong>Layout</strong>: Compute geometry (position, size)</li>
<li><strong>Paint</strong>: Fill in pixels</li>
<li><strong>Composite</strong>: Layer and draw to screen</li>
</ol>
<p>Different properties trigger different stages. The key insight: composite-only animations are cheapest.</p>
<h2>The Golden Properties</h2>
<p>These properties can be animated without triggering layout or paint:</p>
<ul>
<li><code>transform</code> (translate, scale, rotate)</li>
<li><code>opacity</code></li>
</ul>
<p>Everything else is more expensive. <code>width</code>, <code>height</code>, <code>top</code>, <code>left</code> all trigger layout. <code>box-shadow</code> triggers paint. Use them sparingly in animations.</p>
<h2>will-change</h2>
<p>The <code>will-change</code> property hints to the browser that an element will animate. Use it, but do not overuse it — it consumes memory.</p>
<pre><code class="language-css">.animated-element {
  will-change: transform;
}
</code></pre>
<h2>Measuring Performance</h2>
<p>Use Chrome DevTools Performance panel. Look for long frames. A frame should complete in under 16.6ms to maintain 60fps.</p>
<h2>Conclusion</h2>
<p>Fast animations respect the browser. Stick to transforms and opacity, measure your frames, and your users will feel the difference.</p>
